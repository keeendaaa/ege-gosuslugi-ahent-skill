# EGE Gosuslugi Admission Agent Skill

Навык и CLI для оценки текущих шансов поступления по конкурсным спискам Госуслуг. Использует публичный API Вузнавигатора, а не проходные баллы прошлых лет.

## Возможности

- поиск точного `organizationId` по названию и городу;
- исключение филиалов;
- автоматическая поддержка бакалавриата и «Базового высшего образования»;
- произвольные наборы предметов ЕГЭ через `--score`;
- правильный конкурсный балл для каждой программы при наличии запасных предметов;
- бюджет, форма обучения и специалитет как явные фильтры;
- БВИ, согласия, исключённые заявления и приоритет экзаменов при равных баллах;
- каскадный расчет высшего проходного приоритета по всем бюджетным группам вуза;
- пакетный анализ нескольких вузов;
- кликабельные JSON- и PDF-сводки.

Python 3.11 или новее. PDF создаётся через `reportlab`; отдельные системные генераторы PDF не нужны. Для кириллицы автоматически используется Arial на macOS/Windows или DejaVu Sans на Linux.

## Быстрый старт

```bash
python scripts/admission.py report \
  --university "Российский университет транспорта" \
  --university "РГУ им. А.Н. Косыгина" \
  --university "МАДИ" \
  --university "Московский политехнический университет" \
  --city Москва \
  --year 2026 \
  --score "Русский язык=55" \
  --score "Математика=72" \
  --score "Информатика=70" \
  --individual 5 \
  --pdf admission.pdf \
  --json admission.json
```

Можно передать дополнительные результаты, например `--score "Физика=74"`. Для каждой конкурсной группы программа выберет допустимые три экзамена и посчитает соответствующий итог.

## Команды

### Найти вуз

```bash
python scripts/admission.py find --query "МАДИ" --city Москва --year 2026
```

### Получить JSON в терминале

```bash
python scripts/admission.py analyze \
  --org-id 1187 \
  --year 2026 \
  --score "Русский язык=55" \
  --score "Математика=72" \
  --score "Информатика=70" \
  --individual 5
```

### Рассчитать высший проходной приоритет

```bash
python scripts/admission.py priorities \
  --org-id 24 \
  --application-id 1281570 \
  --year 2026
```

Команда загружает все основные бюджетные группы вуза, связывает записи по `idApplication` и распределяет абитуриентов по их приоритетам до устойчивого состояния. Для целевого заявления расчет выполняется и без действующего согласия как гипотетический сценарий. В результате выводятся исходное место среди согласий, эффективное место, ВПП, заполнение группы и диапазон времени обновления списков.

Расчет не объединяет квоты, основной бюджет и платные договоры. Для платного приема используйте `analyze --paid`.

## Как работают приоритеты

Приоритет `1` — самый высокий. Чем больше номер, тем ниже желание поступающего. Согласие на бюджет действует не на отдельную программу, а на все бюджетные конкурсные группы выбранного вуза. Поступающий занимает место только на одном, самом высоком приоритете, куда он проходит после одновременного учета всех списков.

Основные термины:

- **ОВП, основной высший приоритет** — самый высокий приоритет, куда поступающий проходит без учета согласия;
- **ВПП, высший проходной приоритет** — самый высокий приоритет, куда проходит поступающий с действующим согласием; именно по нему происходит зачисление;
- **ВП** — распространенное интерфейсное сокращение высшего приоритета;
- **не ВП** — запись видна в списке, но поступающий сейчас назначен на другой, более высокий приоритет;
- **НП** не является единым федеральным сокращением, поэтому его значение нужно сверять с легендой конкретного вуза.

Если человек проходит одновременно на приоритеты `3` и `4`, он занимает место только на `3`. Из группы `4` его можно убрать лишь после полного расчета, подтверждающего прохождение на `3`. Простое удаление всех людей с более высоким заявленным приоритетом неверно: часть из них может не пройти выше и вернуться в текущую группу.

Правовая основа — пункты 72, 94, 96 и 97 [Порядка приема, утвержденного приказом Минобрнауки России № 821](http://publication.pravo.gov.ru/document/0001202411290031), с изменениями для кампании 2026 года по [приказу № 905](http://publication.pravo.gov.ru/document/0001202511280124).

## Алгоритм каскада

Команда `priorities` использует устойчивое распределение с отложенным принятием:

1. Каждый поступающий с согласием подается в свою активную группу с приоритетом `1`.
2. Каждая группа временно оставляет лучших кандидатов в пределах количества мест.
3. Вытесненные кандидаты переходят на следующий выбранный приоритет.
4. Группы снова оставляют лучших, включая новых кандидатов.
5. Расчет повторяется, пока перемещения не прекратятся.

Полученное состояние удовлетворяет следующим условиям:

- одна бюджетная позиция на человека;
- ни одна группа не превышает вместимость;
- каждый назначен на свой высший доступный приоритет;
- более слабый кандидат не удерживает место вместо более сильного кандидата, который хочет эту группу и не проходит выше.

При одинаковом наборе данных результат не зависит от порядка обработки поступающих.

## Поля публичного API

Для расчета используются:

| Поле | Назначение |
|---|---|
| `idApplication` | связывает одно заявление между конкурсными группами вуза |
| `priority` | выбранный номер приоритета в этой группе |
| `rating` | официальный исходный порядок внутри конкретного списка |
| `consent` | `ONLINE`, `OFFLINE` или `NONE` |
| `statusName` | текущее состояние конкурсной группы заявления |
| `withoutTests` | признак поступления без вступительных испытаний |
| `sumMark` | конкурсный балл |
| `result1`...`result8` | результаты экзаменов в приоритетном порядке |
| `paidContract` | признак платного договора; не является бюджетным согласием |
| `updateDate` | время обновления списка |

Внутри группы используется официальный `rating`, а не повторная сортировка только по `sumMark`. Это сохраняет порядок БВИ и разрешение равенства, уже примененное источником. Если `rating` отсутствует, используется позиция записи в ответе API.

В расчет включаются `ONLINE` и `OFFLINE`. Целевое заявление с `NONE` временно добавляется как сценарий «что будет после подачи согласия». Из расчета исключаются статусы:

- `Конкурсная группа исключена`;
- `Вуз отклонил выбор конкурсной группы`;
- `Ожидаются результаты испытаний`;
- `Отказ от зачисления`;
- `Вы не прошли по конкурсу`.

Одинаковые номера бюджетных приоритетов у одного заявления считаются неоднозначными данными и останавливают расчет вместо неофициального разрешения конфликта.

## Результат `priorities`

На уровне вуза:

| Поле | Значение |
|---|---|
| `consentValues` | значения согласия целевого заявления во всех его группах |
| `hypotheticalConsent` | расчет выполнен как сценарий переноса согласия |
| `highestPassingPriority` | текущий расчетный ВПП |
| `assignedCompetitionId` | группа, которая удерживает целевое заявление |
| `competitionGroups` | количество загруженных основных бюджетных групп |
| `simulatedApplicants` | количество связанных заявлений в модели |
| `updateRange` | самый старый и новый `updateDate` использованного снимка |
| `skippedStatuses` | исключенные записи по статусам |

Для каждой выбранной целевым заявлением группы:

| Поле | Значение |
|---|---|
| `rawRating` | исходное место во всем официальном списке |
| `rawPlaceAmongConsents` | простое место среди согласий до каскада |
| `effectivePlace` | место среди реально назначенных в эту группу; заполнено только для ВПП |
| `assignedCount` | сколько человек модель оставила в группе |
| `vacancies` | незаполненные места после каскада текущего снимка |
| `cutoffRawRating` | исходный рейтинг последнего назначенного человека |
| `isHighestPassingPriority` | является ли группа расчетным ВПП |

## Область применимости

`priorities` рассчитывает основные бюджетные места и главный этап внутри каждого вуза отдельно. Команда не моделирует:

- особую, отдельную и целевую квоты как связанные пулы;
- перенос неиспользованных квотных мест в основной конкурс;
- платное зачисление, договоры и оплату;
- автоматическое распределение между разными вузами;
- еще не опубликованные результаты внутренних испытаний;
- скрытые льготные признаки и правила равенства, отсутствующие в API.

Единого межвузовского порядка предпочтений нет: на бюджете поступающий самостоятельно держит согласие только в одном вузе. Поэтому вузы рассчитываются независимо, а гипотетический перенос согласия является отдельным сценарием.

Списки разных групп могут иметь разные `updateDate`. Чем шире диапазон `updateRange`, тем выше риск, что каскад построен из несинхронного снимка. Результат всегда нужно описывать как текущую оценку, а не гарантию зачисления.

## Что означает результат

«Сейчас внутри бюджета» означает только положение на момент обновления API. Новые согласия могут изменить границу. Команда `priorities` учитывает переходы абитуриентов на более высокие приоритеты внутри одного вуза; обычная команда `analyze` показывает консервативное место без каскадного перераспределения.

## Ограничения

- API принадлежит Госуслугам и может временно ограничивать частые запросы.
- Скрипт не обходит антибот-защиту и не использует авторизацию.
- Точный расчет квот, переносов мест и скрытых правил разрешения равенства может требовать данных, которых нет в публичном API.
